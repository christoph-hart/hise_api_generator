# core.phasor - C++ Exploration

**Base class:** `phasor_base<NV, false>`, `polyphonic_base`
**Classification:** audio_processor

## Signal Path

The phasor outputs a naive 0..1 ramp that wraps via integer truncation (`phase - bitwiseOrZero(phase)`). The output REPLACES channel 0 of the input signal -- `data[0] = (float)phase` in `processFrameInternal()`. Only channel 0 is written; other channels are not modified.

Processing is per-sample on channel 0. `process()` iterates samples on channel 0, casting each to `span<float, 1>` and calling `processFrameInternal()`.

The phasor does NOT use FM modulation (template parameter `useFM = false`).

## Gap Answers

### processing-order: Replace or add?

Replaces. `processFrameInternal()` at line 1266: `data[0] = (float)phase;`. The input signal on channel 0 is overwritten with the ramp value.

### naive-waveform: Is it naive/aliasing?

Yes. The ramp is generated by incrementing a phase accumulator (`uptime += uptimeDelta * multiplier`) and wrapping via integer truncation. There is no anti-aliasing, no PolyBLEP, no band-limiting. At high frequencies, the discontinuity at the wrap point will alias.

### gate-midi-interaction: Does MIDI control Gate?

No. `handleHiseEvent()` (line 1276) only responds to note-on by calling `setFrequency()`. Gate is a manual parameter only. Default is 1.0 (on).

### output-range: What is the output range?

Strictly 0 to <1. The wrapping `phase -= bitwiseOrZero(phase)` removes the integer part, leaving the fractional part. There is a discontinuity at the wrap point (jumps from near 1.0 back to near 0.0).

## Parameters

- **Gate** (0/1): Enables/disables output. Rising edge resets phase to 0.
- **Frequency** (20-20000 Hz): Sets the ramp frequency. Overridden by MIDI note-on.
- **Freq Ratio** (1-16, integer): Multiplier for frequency. Stored in OscData.multiplier.
- **Phase** (0-1): Phase offset stored in OscData.phase.

## Polyphonic Behaviour

`PolyData<OscData, NumVoices> voiceData` provides per-voice phase accumulators.
