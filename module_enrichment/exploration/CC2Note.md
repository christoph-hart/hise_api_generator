# CC2Note (MIDI CC to Note Generator) - C++ Exploration

**Source:** `hi_scripting/scripting/HardcodedScriptProcessor.h` (lines 470-593), `HardcodedScriptProcessor.cpp` (lines 87-105, 243-301)
**Base class:** `HardcodedScriptProcessor` (extends `MidiProcessor`)

## Signal Path

CC2Note intercepts incoming MIDI events to convert CC messages into note triggers for sample-based instruments. It blocks all incoming noteOn/noteOff events (capturing the last note number), then generates new artificial notes whenever the selected CC number is received. The generated note uses the captured note number with the CC value as velocity. Built-in round-robin group cycling selects paired up/down-stroke sample groups on each trigger.

midi in -> [capture & block notes] -> [match CC number] -> [round-robin group select] -> Synth.playNote(lastNote, ccValue) -> midi out

Non-matching CC messages and all other MIDI events pass through unchanged. The matching CC event also passes through (it is not consumed).

## Gap Answers

### cc-to-note-mapping

**Question:** How does CC2Note transform incoming CC messages into note events?

**Answer:** In `onController()` (HardcodedScriptProcessor.h:534-561), the module checks two conditions: `lastNote != -1` (a note is currently held) AND `Message.getControllerNumber() == selectorValue` (CC matches the selected number). When both are true, it calls `Synth.playNote(lastNote, Message.getControllerValue())`. There is no threshold - any CC value (0-127) triggers a note. A CC value of 0 generates a note with velocity 0, which in HISE is typically treated as a noteOff by the synth engine. No explicit noteOff is ever generated by this module; each trigger is a one-shot note-on, making it suited for drum/percussion samples.

### note-number-source

**Question:** How is the generated note number and velocity determined?

**Answer:** The note number comes from `lastNote`, which is captured in `onNoteOn()` (line 521): the last noteOn received before the CC trigger. The velocity comes directly from `Message.getControllerValue()` - the CC value (0-127) maps 1:1 to velocity. The user must hold a note on their keyboard/controller while sending CC messages to trigger sounds. Without a held note (`lastNote == -1`), CC messages are ignored.

### non-matching-cc-passthrough

**Question:** What happens to CC messages that do not match the selected CC number?

**Answer:** Non-matching CCs pass through completely unchanged. The outer `if` in `onController()` (line 536) gates on `Message.getControllerNumber() == selectorValue`, so non-matching CCs skip the entire block. Notably, even matching CC events are NOT consumed - there is no `Message.ignoreEvent(true)` call in `onController()`, so the triggering CC passes through to downstream processors alongside the generated note.

### note-event-passthrough

**Question:** Are existing noteOn/noteOff events passed through?

**Answer:** No. Both `onNoteOn()` (line 522) and `onNoteOff()` (line 527) call `Message.ignoreEvent(true)`, which suppresses the original note events. The module captures the note number from noteOn but blocks the event from reaching the synth. This means the module completely replaces keyboard-driven note input with CC-driven triggering. A keyboard/controller is still needed to set the target note number, but the actual sound is only triggered by the selected CC.

### bypass-active-notes

**Question:** When Bypass is toggled on while a note is active, does the module send a noteOff?

**Answer:** The Bypass parameter does NOT bypass note generation at all. In `onController()` (lines 538-555), the bypass check only gates the round-robin group selection block. The `Synth.playNote()` call at line 559 is OUTSIDE the bypass guard and always executes. When bypass is toggled ON via `onControl()` (line 570), it calls `Sampler.enableRoundRobin(true)` to re-enable the sampler's built-in round-robin. When toggled OFF, it calls `Sampler.enableRoundRobin(false)` so the module's custom group cycling takes over. There is no stuck-note cleanup because bypass never stops note generation. Additionally, since the module only generates note-ons (no note-offs), there are no active notes to clean up in the traditional sense - each trigger is a one-shot.

### description-round-robin

**Question:** Is there actual round-robin logic in the module?

**Answer:** Yes, the module contains genuine round-robin group cycling logic. In `onController()` (lines 540-554):

1. An `updown` boolean alternates on each CC trigger
2. On every other trigger (`!updown`), a new random group index is selected, guaranteed different from the last (`while (thisGroup == lastGroup)`)
3. The final group number is computed as `thisGroup * 2 + (updown ? 1 : 0) + 1`, creating paired groups: (1,2), (3,4), (5,6)...
4. This suggests an up-stroke/down-stroke pairing scheme where each group pair contains alternating articulations
5. `rrAmount` (total group count) is read from `Synth.getAttribute(7)` which corresponds to the parent Sampler's Group Amount parameter
6. `Sampler.setActiveGroup(thisGroupNumber)` sets which sample group plays

The description's mention of "round-robin triggering" accurately reflects built-in functionality. The random selection with no-repeat avoids consecutive identical groups. The `rrAmount / 2` calculation means the module expects groups to be organized in pairs.

## Processing Chain Detail

1. **Note capture & suppress** (negligible): `onNoteOn()` stores note number in `lastNote`, ignores original event. `onNoteOff()` clears `lastNote` if matching, ignores original event.
2. **CC match check** (negligible): `onController()` checks if CC number matches `selectorValue` and `lastNote != -1`
3. **Round-robin group select** (negligible, conditional): When bypass is OFF, alternates up/down state, randomly selects a non-repeating group pair, calls `Sampler.setActiveGroup()`
4. **Note generation** (negligible): `Synth.playNote(lastNote, ccValue)` creates an artificial note event

## Conditional Behavior

- **Bypass = OFF** (value < 0.5): Custom round-robin group cycling is active. Sampler's built-in round-robin is disabled. Groups are selected randomly in pairs with up/down alternation.
- **Bypass = ON** (value >= 0.5): Custom group cycling is skipped. Sampler's built-in round-robin is re-enabled. Notes are still generated from CC.
- **No note held** (lastNote == -1): All CC messages are ignored (no note generation). This is the initial state and the state after all notes are released.
- **CC value = 0**: A note with velocity 0 is generated, which the synth engine typically treats as a note-off.

## Vestigial / Notable

- The Bypass parameter's metadata description ("Bypasses CC-to-note triggering") is inaccurate. It only toggles between custom and built-in round-robin group cycling. Note generation always occurs regardless of bypass state. Flagged in issues.md.
- The `rrAmount` value is only refreshed in `onControl()` (when any UI control is moved), not when the Sampler's group count changes externally. If the group count changes without touching CC2Note's UI, the module may use a stale value.
- The matching CC event is not consumed - it passes through to downstream processors, which could cause unintended side effects.
- Generated notes have no corresponding noteOff. This is by design for one-shot/drum samples but would cause stuck notes with sustaining instruments.

## CPU Assessment

- **Overall baseline**: negligible
- All processing is event-driven (per-MIDI-event, not per-sample or per-block)
- Random number generation and group selection are trivially cheap
- No DSP, no buffers, no per-sample processing

## UI Components

- No custom editor or FloatingTile content types. Uses the default `HardcodedScriptProcessor` editor which renders the Content controls (a button for Bypass and a ComboBox for CC selector).

## Notes

- CC2Note is a `HardcodedScriptProcessor`, meaning it's implemented as a C++ class with script-like callbacks (`onNoteOn`, `onNoteOff`, `onController`, `onControl`) rather than as a traditional C++ MidiProcessor with `processHiseEvent()`. The base class `HardcodedScriptProcessor::processHiseEvent()` dispatches events to these callbacks.
- The module is specifically designed for Sampler-based instruments. It calls `Sampler.enableRoundRobin()` and `Sampler.setActiveGroup()`, which are Sampler-specific API calls. Using it with non-Sampler synths will throw script errors during `onInit()`.
- The default `ccSelector` value of 0 means CC#0 (Bank Select MSB) is the default trigger, which is an unusual choice for percussion triggering. Users will almost certainly need to change this.
- The round-robin pairing scheme (groups 1-2, 3-4, 5-6...) with up/down alternation suggests the module was designed for hi-hat or similar instruments where alternating articulations (open/closed, up/down stroke) need to cycle through multiple velocity layers or variations.
