# Broadcaster -- Project Context

## Project Context

### Real-World Use Cases
- **Centralized event bus**: A large plugin with dozens of scripts defines core broadcasters in a single data layer file. Feature modules subscribe to these central broadcasters rather than depending on each other directly. A single "channel index" broadcaster can have listeners in 10+ separate scripts (mixer, transport, effects, sequencer, waveform display), achieving complete decoupling. This is the dominant architecture for plugins with 20+ scripts.
- **Virtual Control Array (VCA)**: An FX plugin with a parametric EQ uses three coordinated broadcasters to map a single set of UI knobs (Gain/Freq/Q/Mode/Enable) to different EQ bands across multiple processors. One broadcaster watches a radio group to select the active EQ mode, another watches `attachToEqEvents` to update knob values on band selection, and a third watches `attachToModuleParameter` across all EQ processors to keep the knobs in sync with any parameter changes. This allows N bands across M processors to share one set of UI controls.
- **Module parameter UI sync with drag protection**: A synthesizer with interactive envelope editors uses paired broadcasters. A primary broadcaster monitors module parameter changes via `attachToModuleParameter` and updates UI knobs. A companion broadcaster watches mouse events on those same knobs via `attachToComponentMouseEvents` and temporarily bypasses the primary broadcaster during user interaction. This prevents the modulation engine from fighting with user input. After mouse release, a delayed listener re-enables the primary broadcaster.
- **Preset lifecycle coordination**: A plugin with complex preset state uses three broadcasters forming a sequential chain: pre-load (fires before the preset loads, allowing scripts to save lock states), post-load (fires after loading, restoring locked states and rebuilding UI), and preview (managing audition playback). This separates the preset lifecycle into discrete, subscribable events.
- **Lazy loading on first interaction**: A broadcaster attached to root panel mouse hover events triggers deferred sample loading the first time the user interacts with the UI, then permanently bypasses itself via `setBypassed(true)`. This defers heavy initialization until the plugin window is actually opened.

### Complexity Tiers
1. **Single-purpose broadcaster** (most common entry point): One `attachTo*` source, one or two `addListener` targets. Methods needed: `Engine.createBroadcaster`, one attach method, `addListener`. Covers simple radio group page switching, component value monitoring, and context menus.
2. **Multi-target reactive binding**: One source drives multiple target types -- `addComponentPropertyListener` for visual state, `addComponentRefreshListener` for repaints, `addListener` for logic. Methods needed: all of tier 1 plus `addComponentPropertyListener`, `addComponentRefreshListener`, `addComponentValueListener`. Covers parameter-to-UI synchronization and visual state management.
3. **Coordinated broadcaster patterns**: Multiple broadcasters working together -- cross-broadcaster communication via `attachToOtherBroadcaster`, drag-bypass pairs, preset lifecycle chains, and centralized event bus architectures. Methods needed: all of tier 2 plus `attachToOtherBroadcaster`, `setBypassed`, `addDelayedListener`, `setRealtimeMode`, `setEnableQueue`, priority metadata. Covers production-scale plugin architectures.

### Practical Defaults
- Use `attachToRadioGroup` for page navigation -- it handles button state tracking and provides the zero-based selected index directly, eliminating manual radio group management code.
- Use `addDelayedListener` with a 30-100ms delay for debouncing rapid UI filter changes (e.g., search field + tag list + favorites toggling in a preset browser). This prevents expensive operations from running on every keystroke.
- Use the `{ "id": "descriptiveName", "args": ["argName1", "argName2"] }` creation format. It produces readable code and supports metadata tags for the BroadcasterMap debugger.
- Set `priority` in metadata to control execution order when multiple listeners must run in a specific sequence. Higher values execute first. A common pattern is giving state-setting listeners priority 100 and UI-update listeners lower priority.
- Pass `false` (not a function) as the `optionalFunction` parameter in `addComponentPropertyListener` and `addComponentValueListener` for direct mode, where the broadcast value is applied directly without transformation.

### Integration Patterns
- `Broadcaster.attachToRadioGroup()` -> `Broadcaster.addListener()` -> page visibility management -- A radio group broadcaster drives page panel visibility by toggling `visible` properties on page containers in the listener callback. Use `addComponentPropertyListener` with `["visible"]` for the most direct approach.
- `Broadcaster.attachToModuleParameter()` -> `Broadcaster.addComponentValueListener()` -- Module parameter changes drive UI knob values. Use `addComponentValueListener` in direct mode for 1:1 parameter-to-knob mapping, or with a transform callback when the UI scale differs from the parameter range.
- `Broadcaster.attachToComponentMouseEvents()` -> `Broadcaster.setBypassed()` on another broadcaster -- Mouse click detection temporarily suspends a parameter-monitoring broadcaster to prevent feedback during user interaction. Pair with `addDelayedListener` to re-enable after a short delay.
- `Broadcaster.attachToComplexData("AudioFile.Content")` -> `Broadcaster.addListener()` -- Watches audio file loading across multiple player modules and updates UI state (waveform displays, sample name labels, mixer channel icons) reactively.
- `Broadcaster.attachToContextMenu()` -> `Broadcaster.addListener()` -- Right-click context menus with dynamic state functions. The state function controls tick marks and enabled state, and the listener handles the selected action.
- `UserPresetHandler.setPostCallback()` -> `Broadcaster.sendSyncMessage()` -- A preset post-load callback triggers a broadcaster to notify all subscribed modules that the preset state has changed, without those modules needing to know about the preset system directly.

## Common Mistakes
| Wrong | Right | Explanation |
|-------|-------|-------------|
| Adding listeners in many separate scripts that directly call each other's functions | Defining core broadcasters in a central data file and subscribing from feature modules | Direct function calls between scripts create tight coupling. Broadcasters decouple event producers from consumers. A single broadcaster can notify 10+ scripts without any script knowing about the others. |
| Using `Content.getComponent()` inside a broadcaster callback to look up components on every fire | Caching component references in `const var` at init time and passing them via closure or the `object` parameter | Broadcaster callbacks can fire frequently (mouse events, parameter changes). Repeated `Content.getComponent()` calls are unnecessary overhead when references can be cached once at initialization. |
| Using `setControlCallback` for all UI interactions in a large project | Using `setControlCallback` for simple value-to-parameter mapping, but Broadcasters for cross-cutting concerns (preset lifecycle, page navigation, parameter monitoring, context menus) | `setControlCallback` is fine for direct 1:1 control-to-parameter wiring. Broadcasters add value when multiple unrelated systems need to react to the same event, or when reacting to events that `setControlCallback` cannot handle (module parameter changes, EQ events, complex data, mouse events, etc.). |
