# Forum Insight Distribution Guide

How to distribute forum-derived insights (gaps, warnings, tips, patterns) into the correct documentation files across all enrichment domains. This guide is used by the Step 4b backport triage agent and by authors integrating forum findings into new pages.

All paths relative to `HISE/tools/api generator/`.

---

## Core Principle

**Never create a "Warnings" or "Tips" section.** Distribute each insight to the most relevant location in the existing page. Users discover the warning when they're reading about the thing that has the gotcha.

---

## 1. Source File Mapping (write targets)

Each domain has specific source files that are the single source of truth. Never write to generated artifacts.

| Domain | Website source | MCP/AI source | File pattern |
|---|---|---|---|
| **Modules** | `module_enrichment/pages/` | (llmRef in same frontmatter) | `{ModuleId}.md` |
| **Scriptnode** | `scriptnode_enrichment/output/` | (llmRef in same frontmatter) | `{category}/{nodeId}.md` |
| **Scripting API** | `enrichment/phase4/auto/{ClassName}/` | `enrichment/phase4b/{ClassName}/` | `{method}.md` + `Readme.md` |
| **UI Components** | `ui_enrichment/pages/components/` | (in same file) | `{Component}.md` |
| **UI Floating Tiles** | `ui_enrichment/pages/floating-tiles/` | (in same file) | `{Tile}.md` |
| **Language** | `language_enrichment/output/` | - | `{language}.md` |

### Validation gate

Before writing to any file, verify:
1. The path matches one of the source patterns above
2. The path is NOT inside any of these generated artifact directories:
   - `enrichment/output/` (generated by `api_enrich.py preview`)
   - `enrichment/phase4b/` is NOT generated - it IS a valid write target for Scripting API MCP docs
   - website `content/v2/` (generated by `publish.py`)
   - `forum_cache/` (throwaway)
3. The file already exists (do not create new pages for cross-domain insights - flag for later)

### Scripting API dual-write rule

The Scripting API is the only domain with split sources. When adding a forum insight:
- **`enrichment/phase4/auto/{ClassName}/{method}.md`** - website-facing prose (userDocs)
- **`enrichment/phase4b/{ClassName}/{method}.md`** - MCP server reference (llmRef)

Both must be updated. The website prose can be longer/richer; the MCP reference should be concise.

Verification flow:
1. Look up the exploration file for the target
2. Search it for the specific claim (grep for keywords)
3. If found and confirms/denies the claim: use that as evidence, no `verify-forum-claim` needed
4. If not found or inconclusive: run `verify-forum-claim` for targeted C++ verification

## 3. Insight Classification Rules

Each forum insight maps to exactly one placement type. Classify before writing.

### A. Parameter hint (inline in parameter/modulation table)

**When:** The insight is about a specific parameter's behaviour, limitation, or edge case.

**MDC syntax** (new `hints` field on parameter rows):
```yaml
::parameter-table
---
groups:
  - label: Envelope
    params:
      - name: FadeIn
        desc: "Fade-in time after note trigger..."
        range: "0 - 3000 ms"
        default: "1000 ms"
        hints:
          - type: warning
            text: "Has no effect when the LFO is in a Container's modulation chain."
---
::
```

Same syntax works for `::modulation-table` rows:
```yaml
::modulation-table
---
chains:
  - name: "LFO Intensity Mod"
    desc: "Scales the LFO output depth..."
    scope: "monophonic"
    constrainer: "Any"
    hints:
      - type: warning
        text: "In pitch mode, reducing intensity shifts the pitch center asymmetrically."
---
::
```

Hint types: `warning` (amber, triangle-alert icon) or `tip` (blue, lightbulb icon).

**Keep hints to 1-2 sentences.** If it needs more explanation, put the detail in the Notes section and add a brief hint pointing to it.

Hints support rich formatting: use `backticks` for API names, parameters, and code values, and **bold** for emphasis. This helps users scan hints quickly.

### B. Notes section prose

**When:** The insight is about a behaviour mode, general workflow guidance, or a limitation not tied to a single parameter.

Add 1-3 sentences to the existing Notes section at the relevant paragraph. Do not create a new subsection - weave it into the existing prose naturally.

Examples:
- TempoSync range limited to 1/1 bar -> add after the tempo sync paragraph
- Dual reference requirement for TableProcessor + Modulator -> add after the interface usage paragraph

### C. commonMistakes frontmatter

**When:** The insight is a clear "wrong vs right" pattern that users hit repeatedly (3+ independent forum threads).

```yaml
commonMistakes:
  - title: "Short descriptive title"
    wrong: "What users incorrectly do"
    right: "What they should do instead"
    explanation: "Observable consequence of the wrong approach"
```

Every entry MUST have a `title` field. publish.py renders commonMistakes as `::common-mistakes` MDC components.

### D. Cross-domain routing

**When:** The insight is really about a different module, API class, or node than the one the forum search targeted.

Route it to the correct domain's source file. Common cross-domain patterns:

| Forum target | Insight about | Route to |
|---|---|---|
| Convolution module | `Engine.loadAudioFilesIntoPool()` | `enrichment/phase4/auto/Engine/loadAudioFilesIntoPool.md` |
| Convolution module | `AudioSampleProcessor.setFile()` | `enrichment/phase4/auto/AudioSampleProcessor/setFile.md` |
| LFO module | `Modulator.getCurrentLevel()` | `enrichment/phase4/auto/Modulator/getCurrentLevel.md` |
| LFO module | Filter aliasing with LFO | `module_enrichment/pages/PolyphonicFilter.md` |

For cross-domain insights, add the target to the page's See Also section (if not already present) rather than cluttering the Notes with cross-references.

### E. Skip

**When:** The insight is too general, already documented, or not actionable for a specific page.

Skip candidates:
- Feature requests that haven't been implemented
- Bugs that have been fixed in recent versions (unless the fix needs documentation)
- General scripting patterns not tied to a specific API surface
- Insights already covered by the existing page content

## 4. Distribution Workflow

### Hard rule: write before delete

When an insight is classified as cross-domain (type D), the target file MUST be written before removing the insight from the source page. Never delete content from one page without first confirming it exists in its new location. The workflow enforces this order.

### Per-insight workflow

For each forum insight in `forum/insights/{Target}.json`:

1. **Classify** - Determine placement type (A-E above)
2. **Verify** - If `verify: true`, check exploration data first, then `verify-forum-claim` if needed
3. **Write to target** - For cross-domain (type D): write to the target file first
4. **Write to source** - For types A/B/C: add hint, prose, or commonMistake to the source page
5. **Cross-reference** - For type D: add the target to the source page's See Also section (if not already present)
6. **Update llmRef** - For modules/scriptnode, update the llmRef frontmatter field to reflect the new content
7. **Dual-write** - For Scripting API targets: update both phase4/auto and phase4b files

For each forum gap in `forum/gaps/{Target}.json`:

1. **Check coverage** - Does the existing page already answer this question?
2. **If covered** - Mark as covered, no action needed
3. **If answerable from exploration** - Extract the answer, add to the page (Notes or parameter description)
4. **If not answerable** - Flag for targeted C++ verification or note as a gap for future exploration

## 5. Examples

### Parameter hint (LFO FadeIn)

Before:
```yaml
- name: FadeIn
  desc: "Fade-in time after note trigger..."
  range: "0 - 3000 ms"
  default: "1000 ms"
```

After:
```yaml
- name: FadeIn
  desc: "Fade-in time after note trigger..."
  range: "0 - 3000 ms"
  default: "1000 ms"
  hints:
    - type: warning
      text: "Has no effect when the LFO is in a Container's modulation chain. Place inside the sound generator or use a Global LFO."
```

### Notes section prose (TempoSync limit)

Before:
> The frequency modulation chain is heavily downsampled...

After:
> The frequency modulation chain is heavily downsampled...
>
> The built-in TempoSync parameter supports divisions from 1/32T up to 1/1 bar. Slower rates (2/1, 4/1) are not available without modifying the HISE source code. For slower tempo-synced modulation, use a scriptnode-based LFO with a tempo node that supports a multiplier.

### Cross-domain (Convolution -> Engine.loadAudioFilesIntoPool)

1. Write to `enrichment/phase4/auto/Engine/loadAudioFilesIntoPool.md` first:
   > Add pitfall about this being the #1 recurring issue for Convolution Reverb users.

2. Write to `enrichment/phase4b/Engine/loadAudioFilesIntoPool.md`:
   > Add concise anti-pattern note for MCP consumers.

3. Add `Engine.loadAudioFilesIntoPool` to the Convolution page's See Also section (if not already present).
